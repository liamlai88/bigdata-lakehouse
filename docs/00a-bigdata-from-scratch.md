# 阶段 0a：大数据是什么 —— 从零讲起

> 目标读者：完全没接触过大数据。读完应该能回答：
> "为什么不能用 MySQL 解决一切？为什么大数据要那么多组件？"

---

## 1. 一切的起点：数据放不下 + 算不动

你写一个普通网站，所有数据存在 **MySQL**。这套东西在数据量小的时候完美：

- 用户表 100 万行 → 一台机器轻松
- `SELECT * FROM users WHERE id=123` → 1 毫秒返回

**问题来了**，假设你的网站火了：

| 场景 | MySQL 还行吗？ |
|---|---|
| 用户表 1000 万行 | ✅ 还行 |
| 用户行为日志 100 亿行 | ❌ 单机磁盘装不下 |
| "算一下过去 30 天每个国家的 GMV" 扫 50 亿行 | ❌ 跑半天，把数据库拖垮 |
| 一边在线交易一边跑大报表 | ❌ 互相干扰，业务卡死 |

**大数据这个学科，本质上就在解决两个问题**：

1. **数据放不下** → 怎么把数据切开存到很多台机器上
2. **算不动** → 怎么让很多台机器一起算

所有你听过的"大数据组件"（Hadoop、Spark、Flink、Hive...），全是围绕这两件事造的轮子。

---

## 2. 第一个核心思路：分而治之

> 一个人搬不动 1 吨砖，但 100 个人每人搬 10 公斤就可以。

**数据切开** = 把一张 100 亿行的大表，切成 1000 份，每份 1000 万行，分别放在 1000 台机器上。

**计算切开** = "算每个国家的 GMV" 这件事，让 1000 台机器各算自己那 1000 万行的小结果，最后汇总一下。

这个思路 2004 年 Google 写了三篇论文（GFS、MapReduce、BigTable）讲清楚了，然后开源社区照着抄了一套：

| Google 论文 | 开源实现 | 干什么 |
|---|---|---|
| GFS | **HDFS** | 分布式文件系统：把文件切块放多台机器 |
| MapReduce | **MapReduce** | 分布式计算：很多机器一起算 |
| BigTable | **HBase** | 分布式 KV 数据库 |

这套东西打包起来叫 **Hadoop**，是大数据的"上古时代"。

---

## 3. OLTP vs OLAP：两个世界

大数据要先认这一对词，所有后面的分类都从这里来。

| | OLTP（在线交易） | OLAP（在线分析） |
|---|---|---|
| 中文 | 联机事务处理 | 联机分析处理 |
| 典型场景 | 下单、转账、注册 | 报表、看板、用户画像 |
| 一次操作改/查多少行 | 改 1 行、查几行 | 不改，**扫几百万行算汇总** |
| 要求 | 快、准、不丢、并发高 | 吞吐大、能扫大数据，慢一点没关系 |
| 代表产品 | MySQL、PostgreSQL、Oracle | Hive、ClickHouse、Doris、Spark、Trino |
| 存储方式 | **行存** | **列存** |

**记住一句话**：大数据 99% 在做 OLAP。
MySQL 是 OLTP 之王，但拿它做 OLAP（扫几十亿行算报表）就是用菜刀砍树。

---

## 4. 行存 vs 列存：第二个核心思路

这是大数据里最值得理解的概念，跟所有后面的组件都有关。

假设有一张表：

| user_id | country | age | amount |
|---|---|---|---|
| 1 | US | 25 | 100 |
| 2 | CN | 30 | 200 |
| 3 | US | 28 | 150 |

**行存**（MySQL 这么存）：磁盘上按行一个挨一个
```
[1, US, 25, 100] [2, CN, 30, 200] [3, US, 28, 150]
```
查 `SELECT * FROM t WHERE id=2` → 直接读第二行，超快。
查 `SELECT SUM(amount) FROM t` → 也得把每行读出来，把不要的 user_id/country/age 都过一遍，**浪费 75% 磁盘 IO**。

**列存**（大数据这么存）：磁盘上按列一个挨一个
```
user_id 列: [1, 2, 3]
country 列: [US, CN, US]
age     列: [25, 30, 28]
amount  列: [100, 200, 150]
```
查 `SELECT SUM(amount)` → **只读 amount 这一列**，IO 直接省 75%。
查 `SELECT * WHERE id=2` → 要拼四列，反而慢。

**结论**：
- 改一行 / 查一行 → 行存（OLTP）
- 扫几百万行算汇总 → 列存（OLAP）

后面你会反复遇到的 **Parquet** 文件格式，就是列存。

## 5. 大数据组件全景图（这次不要被吓到）

听过的那一堆名字，其实就是**数据从产生到看到看板**这条流水线上每一段的工具：

```
   ① 数据产生         ② 传输/采集        ③ 存储          ④ 计算         ⑤ 查询      ⑥ 展示
                                                          ↓                ↓
   App 埋点  ──►  Kafka / Flume   ──►  HDFS / S3   ──►  Spark        Hive       Superset
   交易数据  ──►  Debezium (CDC)        MinIO              (批处理)    (SQL on    BI 看板
   日志文件                              Parquet 文件       Flink         数据)
                                                          (流处理)
                                                                       Trino /
                                                                       ClickHouse
                                                                       (查得快)
                                       ┌──────────────┐
                                       │ 表格式：Iceberg│ ← 给上面的 Parquet 文件
                                       │  Hudi / Delta │   加上"表"的能力（ACID、Schema 演进）
                                       └──────────────┘

                                       ┌──────────────┐
                                       │ 调度: Airflow │ ← 谁先跑谁后跑、几点跑、失败重试
                                       └──────────────┘
```

每一段都有"为什么需要"：

| 段 | 为什么需要 | 不用会怎样 |
|---|---|---|
| ② Kafka | 数据产生方和消费方解耦、削峰 | 100 万用户同时下单 → 直接打挂下游 |
| ③ HDFS / S3 | 一台机器存不下 | 数据爆炸 |
| ③ Parquet | OLAP 要列存 | 扫数据慢 10 倍 |
| ③ Iceberg | 让"一堆 Parquet 文件" 像"一张表"（能改、能演进 schema） | 删一行都很痛苦 |
| ④ Spark | 把计算分到 100 台机器 | 单机算 30 天数据要算一周 |
| ④ Flink | 数据一来就算（不用等 T+1） | 实时看板永远落后一天 |
| ⑤ Trino | 让分析师直接写 SQL，秒级返回 | 每次问问题都要工程师写 Spark 作业 |
| ⑥ Superset | 让运营 / 老板看图，不写 SQL | 大家排队问数据 |
| 调度 Airflow | 100 个作业有依赖、要按时跑 | 凌晨手动 crontab，挂了没人知道 |

## 6. 批处理 vs 流处理：第三个核心区分

| | 批处理 (Batch) | 流处理 (Stream) |
|---|---|---|
| 数据怎么来的 | 攒一坨，定时算（每天凌晨算昨天） | 来一条算一条 |
| 延迟 | 小时 / 天级 | 秒 / 毫秒级 |
| 典型代表 | Spark、Hive、MapReduce | Flink、Spark Streaming |
| 典型场景 | 离线日报、月报、模型训练 | 实时大屏、风控告警、实时推荐 |

**一个比喻**：
- 批处理 = 邮局每天下午统一收信、统一送
- 流处理 = 微信，发一条到一条

现代公司两个都要：**离线批处理**算昨天的稳定报表，**实时流处理**算"过去 5 分钟"。
这就叫 **Lambda 架构**（有点过时）或 **流批一体**（新趋势，Flink 主推）。

## 7. 数据仓库 vs 数据湖 vs 湖仓一体

这三个词阿里云、AWS 满天飞，必须搞清楚。

**数据仓库（Data Warehouse）**：老派做法
- 数据必须**先清洗、先建好 schema** 才能进来（Schema-on-Write）
- 例子：Teradata、阿里 MaxCompute、Snowflake、Hive
- 优点：查得快，治理好
- 缺点：贵，灵活性差，存不了图片/视频/JSON

**数据湖（Data Lake）**：新派做法（约 2015 年）
- 啥都先扔进来（Schema-on-Read，要查的时候再解析）
- 例子：HDFS 一堆文件、S3 一堆 JSON
- 优点：便宜、什么都能存
- 缺点：很快变"数据沼泽"，没人知道哪个文件是什么

**湖仓一体（Lakehouse）**：最新方案（约 2020 年）
- 在数据湖（S3 + Parquet）上面加一层"表格式"（Iceberg / Hudi / Delta）
- 让数据湖也能像数仓一样有 ACID、Schema、时间旅行
- 例子：Databricks、阿里 EMR + DLF、AWS S3 + Iceberg
- **本项目走的就是这条路**

一张图：
```
传统数仓:  [结构化数据] → [Hive/MaxCompute] → 报表
数据湖:    [啥都行]    → [S3 一堆文件]    → 自己 Spark 算
湖仓一体:  [啥都行]    → [S3 + Iceberg]   → 像数仓一样查 + 像湖一样灵活
              ↑
           我们要做的
```

## 8. 这个项目你要学的东西，一句话

**我们要在自己的 Mac 上，搭一个"小型阿里云"**：
- 用 **MinIO** 当 S3（对象存储）
- 用 **Parquet + Iceberg** 当表（湖仓核心）
- 用 **Spark** 跑昨天的批量计算
- 用 **Flink** 跑实时的流计算
- 用 **Trino** 提供 SQL 查询入口
- 用 **Airflow** 调度
- 用 **Superset** 看图
- 最后用 **ai-gateway** 把"运营说人话"翻译成"Trino SQL"

跑完一遍，你会对**阿里云 EMR / DLF / Hologres / MaxCompute / Flink / Quick BI / DataWorks** 这套产品矩阵自然就懂了——它们就是上面这些开源组件的商业化包装。

## 9. 你需要先记住的 8 个词（其他后面再学）

| 词 | 一句话 |
|---|---|
| **OLTP** | 改/查一行的数据库（MySQL） |
| **OLAP** | 扫一大堆行算汇总（Hive / Spark） |
| **行存 / 列存** | 数据在磁盘上按行排还是按列排 |
| **批处理 / 流处理** | 攒一坨再算 / 来一条算一条 |
| **数据湖** | S3 上扔一堆文件 |
| **湖仓一体** | 数据湖 + 表格式（Iceberg）= 又便宜又能 SQL |
| **ODS/DWD/DWS/ADS** | 数仓分层，原始 → 明细 → 聚合 → 给应用 |
| **Kafka** | 消息队列，数据传输的"快递中转站" |
